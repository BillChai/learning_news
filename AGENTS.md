# AGENTS.md — 每日 IT 新聞摘要自動化指令

這份文件是本 repo 的**唯一事實來源**。任何 AI agent（不限廠商）只要具備下方「必要能力」，
讀完這份文件就能獨立重現「讀偏好 → 選文 → 開 PR → 回饋寫回」的完整流程，不需要額外背景知識。

## 0. 必要能力（廠商中立）

執行這份文件前，確認你（執行的 AI agent）具備：

1. **網路搜尋 / 讀取網頁**的能力（不限於特定搜尋工具，只要能查到近期文章與其網址即可）。
2. 對這個 GitHub repo（`BillChai/learning_news`）的**讀取檔案、新增分支、寫入檔案、開啟 PR**
   的能力（不限於特定 MCP 工具或 CLI，只要能完成這些 GitHub 操作即可）。

如果你不具備第 2 項能力，停止並回報「缺少 GitHub 寫入權限」，不要嘗試用其他方式繞過
（例如只在對話裡輸出內容卻假裝已經開了 PR）。

## Step 0：載入情境

先完整讀取本 repo 根目錄的 `preferences.md`。裡面記錄了過去使用者對主題/來源的喜歡/不喜歡
回饋，之後每一步的主題比例與來源選擇都要依它調整。

**若讀不到這個檔案（不存在或讀取失敗）**：停下並回報「讀不到 preferences.md」，不要用預設值
默默繼續執行（避免產出一份完全沒對齊使用者偏好的摘要）。

## Step 1：主題配額表

目標 15 則，除非 `preferences.md` 明確要求調整比例，否則依下表分配（用這張表逐項核對數量，
不要用「大概分配」的方式判斷）：

| 類別 | 數量 | 說明 |
|---|---|---|
| 工程知識 / 教學型 | 5 | 有教育性、值得學習的內容：好用工具/函式庫/技巧、原理科普（「這個東西怎麼運作」）、best practice、深入淺出的技術解析。不是單純新聞快訊。 |
| 知名工程師個人部落格近期文章 | 1–2 | 例如 Simon Willison、Julia Evans、Dan Abramov、Martin Fowler、Charity Majors、Kent Beck、Werner Vogels、Armin Ronacher、Dan Luu、antirez、fasterthanlime。挑近期有更新的即可，不用每個都查。`preferences.md` 裡若有使用者特別喜歡的工程師/部落格，優先納入。 |
| GitHub Trending 熱門 repo | 1–2 | 近期 star 數明顯上升、跟後端/AI/工具鏈相關。避免純娛樂/跟工程無關的 repo。 |
| 工作技術棧 + 更廣泛後端主題（合併同一桶） | 剩餘名額 | 這兩個子領域**共用**剩餘名額，不需要在兩者之間平均分配，依當天實際找到的內容比例分配即可：工作技術棧＝Kotlin/Java（後端）、AWS/AWS CDK（IaC）、OpenAPI/Swagger；更廣泛後端主題＝AI/LLM 工程應用、資料庫（DB）、分散式系統/系統設計/架構、綜合 IT 新聞熱門話題。 |

## Step 2：優先來源清單（先查這些，開放搜尋是備援）

**規則：先查下面這份清單裡的來源；只有這份清單找不到足夠內容時，才用開放式網路搜尋補足。**
不論來自清單還是開放搜尋，最終每一則都要通過 Step 5 的逐條檢查表——開放搜尋結果**不能**
因為「補充」的身份就跳過檢查。

這條規則是拉齊不同 AI 之間選文品質最重要的一條：不要倚賴你自己對「什麼是好來源」的隱性判斷，
優先信任這份清單。

- Hacker News（news.ycombinator.com）
- InfoQ
- AWS What's New / AWS Blog
- Postgres Weekly
- High Scalability
- The Pragmatic Engineer
- Latent Space
- `preferences.md` 裡使用者明確指定的其他來源（若有，優先度高於本清單）

## Step 3：禁止主題 / 內容型態

一律排除，不要收錄：

- Android、iOS、Swift 的**版本更新 / SDK 更新 / release notes**類新聞，除非是極少數影響整個
  產業的重大事件。
- 純行銷公告文（廠商自吹自擂、沒有技術細節）。
- 「2026 年 N 大工具/趨勢」這類空洞列表型 SEO 文章（沒有具體技術內容，只是關鍵字堆疊）。
- `preferences.md` 裡使用者明確標示「不喜歡」的主題或來源。

## Step 4：搜尋時間窗

優先找近 1–3 天內的最新文章/部落格/新聞；若當日內容不足，可放寬到近一週，但要在該則摘要裡
誠實標註「較早」或給出實際發布日期。找不到足夠新內容時要誠實說明，不要為了湊滿 15 則硬塞
不相關或過舊的內容。

## Step 5：機械式逐條檢查表（15 則全部要過，選文完成後再逐一核對一次）

對每一則，逐項確認：

1. 這個網址是不是指向**一篇具體文章 / 一個具體 repo**，而不是列表頁、首頁、tag 頁、
   搜尋結果頁？（例如：GitHub Trending 項目要給 `github.com/owner/repo`，不能給
   `github.com/trending`；部落格要給該篇文章網址，不能給部落格首頁；新聞要給該則報導本身，
   不能給 news archive 或 HN 首頁。）
2. 打開這個網址，內容是不是對應**單一一篇**有明確標題與發布日期的內容，而不是並列多篇標題
   的列表？若頁面只有「更新日期」沒有明確發布日期，用更新日期替代即可，並在摘要裡照實標註。
3. 是否已經在最近 14 天的 `digests/*.md` 裡出現過同一篇文章或同一個 repo？（見 Step 6）

只要任何一項回答「不符合」，就直接換掉這一則、重新找一則替代，**不要**為了保留這則而硬凹
「其實這樣也算符合」。

**追溯要求**：在你回報給使用者、以及寫進 PR 說明時，每一則額外註明「屬於哪個配額類別」與
「來自優先來源清單，還是開放搜尋補充」。這是為了讓人在 review PR 時，能回頭核對你是否真的
做過上面的檢查，而不只是宣稱做過。

## Step 6：去重規則

選文前，先列出本 repo `digests/` 目錄下最近 14 天的檔案（依檔名 `YYYY-MM-DD.md` 判斷日期），
讀取其內容，確認今天要選的 15 則裡沒有任何一則是重複選過的文章或 repo。

## Step 7：輸出格式（`digests/YYYY-MM-DD.md`）

以當天日期（`YYYY-MM-DD`，使用者本地時區）為準，寫入 `digests/YYYY-MM-DD.md`，格式固定如下,
不要自行更改結構：

- 檔案開頭：`# IT 新聞日報 YYYY-MM-DD（週X）`，接一段簡短的主題分布總覽。
- 依 Step 1 的類別分組，每則包含：
  - 編號
  - 主題標籤（如 `[工程知識]`、`[Blog]`、`[GitHub Trending]`、`[AI]`、`[DB]`、`[AWS]` 等，
    可疊加多個標籤）
  - 標題（粗體）
  - 一句話摘要，說明為何值得看
  - 來源名稱與連結（必須是 Step 5 檢查過的具體文章/repo 連結）
  - 發布日期（若較早於搜尋時間窗，要標註「較早」）
- 全篇使用**繁體中文**（技術名詞、產品名、程式碼保留原文）。

## Step 8：開分支與開 PR

1. 檢查 `digest-YYYY-MM-DD` 這個分支名今天是否已經存在（代表今天已經有人跑過一次）。若已存在，
   改用 `digest-YYYY-MM-DD-2`、`-3`……依序遞增，**不要**覆蓋既有分支或既有 PR。
2. 從 `main` 建立分支，寫入 `digests/YYYY-MM-DD.md`。
3. 開一個 PR：head 為上一步的分支，base 為 `main`，title 例如「IT 新聞日報 YYYY-MM-DD」，
   body 簡述本次涵蓋的主題分布（含每則的類別/來源追溯，見 Step 5）。
4. **不要自動 merge** 這個 PR，留給使用者自己決定要不要合併。

## Step 9：在對話中呈現並邀請回饋

把同一份 15 則清單也用繁體中文呈現在對話裡（技術名詞/產品名保留原文），附上 PR 連結。呈現
完畢後明確邀請使用者給回饋，例如：「喜歡第幾則/哪個主題，不喜歡哪個主題或來源」。

## Step 10：回饋寫回 `preferences.md`

**觸發條件**：只有在使用者於對話中，針對這次呈現的內容，明確給出喜歡/不喜歡的回饋時才觸發
這一步。沒有回饋就不用做這一步。

1. 讀取 `preferences.md` 目前版本（讀 Step 8 用的那個分支上的版本；若該分支還沒有這個檔案就
   讀 `main` 上的）。
2. 在同一個分支上更新 `preferences.md`，在「回饋紀錄」區塊**追加**一筆（不要覆蓋舊紀錄，
   追加在區塊最後）,**固定使用下面這個格式**（舊紀錄格式不一致，是歷史遺留，之後統一按這個
   格式寫,不要模仿舊格式的變體）：

   ```
   - YYYY-MM-DD：[喜歡/不喜歡] 主題或來源名稱——原因（若使用者沒說明原因，寫「未說明」）。
   ```

   例如：`- 2026-07-17：[不喜歡] AWS 週報彙整型內容——太雜訊,想看更聚焦的單一主題。`

## Step 11：失敗與例外處理

- 找不到滿 15 則符合品質標準的內容時：**發布較少則數，並在 PR 說明裡明確註明「本次僅 N 則，
  原因是……」**，不要為了湊數硬塞不合格內容。
- 若第 0 節的 GitHub 寫入能力在執行中途失效或被拒絕：回報「GitHub 寫入權限不足，在第 X 步
  失敗」，不要假裝已完成。
- 若同一天已經有其他 agent 執行過（Step 8 發現分支已存在）：正常用遞增尾碼繼續執行，不用
  停下來問使用者，但要在 PR 說明裡註明「這是今天第 N 次執行」。
