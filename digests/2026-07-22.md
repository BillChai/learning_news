4. **[工程知識/JVM] ZGC vs G1GC in Java 26：到底該用哪個？**
   對比 JVM 兩大 GC：G1 是分 region、Garbage-First、以「可預測的暫停時間＋吞吐量」為目標；ZGC 幾乎全並行、暫停控制在微秒級。實測到 p99 兩者延遲幾乎相同，但 p999/p9999 尾延遲 ZGC 明顯勝出（G1 >20ms vs ZGC ~50µs）。後端跑 JVM（Kotlin/Java）調校與選型的實用依據。
   來源：Java Code Geeks — <https://www.javacodegeeks.com/2026/06/zgc-vs-g1gc-in-java-26-which-gc-should-you-actually-use.html>｜發布：2026-06（較早）

## AWS

10. **[AWS] AWS Lambda 推出 self-managed code storage（自管程式碼儲存）**
    Lambda 現可把程式碼直接放在你自己的 S3 bucket，設定 `S3ObjectStorageMode=REFERENCE` 讓 Lambda 直接引用而不再複製一份中介副本——移除 75GB 儲存上限（Lambda 託管上限也從 75GB 提升到 300GB/region）、縮短函式啟用時間，且只付標準 S3 費用。部署大量函式/layer 的團隊直接受惠，全商用區可用。
    來源：AWS What's New — <https://aws.amazon.com/about-aws/whats-new/2026/07/lambda-self-managed-code-storage/>｜發布：2026-07-15

11. **[AWS/架構] Removing a Hidden Round Trip from a Multi-Region AWS API**
    實戰案例：某 settings 服務原本每次認證請求前都要先打一通「region discovery」（因為 SigV4 把簽章綁死在單一 service+region），同區帶來 p90 75–100ms、跨洲最壞近 1 秒延遲。改用 SigV4a（可對「一組 region」簽章）後，簽一次、路由交給基礎設施，消掉那趟多餘往返。程式改動不大，但花了半年協調相依團隊。談「當初的設計不是錯，是後來條件變了才值得遷移」的取捨很到位。
    來源：InfoQ（Suresh Gururajan）— <https://www.infoq.com/articles/aws-multi-region-signing/>｜發布：2026-07-13
